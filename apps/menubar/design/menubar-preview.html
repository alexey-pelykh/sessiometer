Sessiometer — menubar popover · all 9 states

The macOS status-bar popover across every launch-or-attach state, at true 380 px width, in the native macOS language — vibrancy, meters, drawn sparklines, label-primary identity. Each state is a distinct panel message + affordance under the shared 4-state glance glyph (#524); the menubar never renders healthy on a degraded daemon. A visual build-reference for the SwiftUI panel (#168/#169) — it does not port to code.

Design notes — the 9-state map in the native macOS language

  • One surface, nine states, one language. Every state is a render of the same 380 px vibrancy popover — healthy rosters and degraded messages share the frosted material, hairline dividers, SF-symbol-style glyphs and tabular numerals. Each state is a distinct icon-shape + panel message + affordance (the 9-state map); the caption glyph above each card is that state’s menubar shape.
  • Healthy = the status render. Status and Stats share one language: label-primary two-line identity (Work / Personal / Scratch + monogram badge, provider below), session & weekly usage meters on the status CLI’s own bands (green <75 / amber 75–89 / red ≥90) with resets in its compact duration form (2h14m, 3d), the smart swap hero callout, and drawn SVG sparklines with the neutral symmetric SIGNAL band. Every percent carries its band in colour — a healthy one included — exactly as the CLI paints it.
  • Never healthy on a degraded daemon (#137). Disconnected shows the last-known roster dimmed + “stale” with a broken-link badge and a Reconnect escape — never frozen-as-live; swaps paused. Crash-looping persists a fault shape and holds status until a fresh snapshot survives a stability window (the healthy-flash is debounced).
  • Freshness is a spectrum. Stale-snapshot is milder than disconnected — still connected, so the full roster renders live with only a subtle amber stale 6m modifier in the header and an amber freshness line in the footer; the swap stays enabled (the daemon re-mints viability on swap).
  • Blocked states are honest and non-intrusive. Keychain-locked is a fleet-wide daemon fault — the shared login keychain is locked, so the daemon can’t read any account’s credential and no per-row auth cell reflects it: the roster stays a full healthy green while the fault rides the strip under the header, the paused swap callout, and the “only waits, never prompts” footer. That line is a contract: the app never prompts or auto-unlocks. Version-skew refuses to render any % and offers a copy-command instead.
  • Bounded blindness is a per-row MODIFIER, not a tenth state (#479/#485). When the active account’s /usage poll is rate-limited (429), the daemon holds a bounded anchor (ADR-0017) and pushes blind_active on an otherwise connected snapshot. That row’s two live meters collapse into one HELD block — a dashed last-known session bar (never a live fill — #137), a blind {dur} chip, a LAST-KNOWN · RATE-LIMITED caption, and the auto-protection verdict — while the header and footer stay fresh: that locality is exactly the tell separating it from a whole-snapshot stale. The health slot takes an eye-slash (usage visibility lost — a distinct shape, never colour-alone). OK stays calm and the menubar glance keeps reading healthy (bounded, self-resolving — no cry-wolf); DEGRADED (the gate armed but acting on a STALE anchor) gains an at-risk orange leading rule + orange eye-slash + orange verdict, and escalates the glance to attention. Treatment per the owner-ratified record (hq strategy/design-menubar.md §blind, ef1a240): the ratified render itself was never committed, so these frames are the oracle standing in for it — the record, not the shipped code, is their authority.
  • Capture is a real action; copy-command only where the app can’t act. First-run / empty-roster onboarding captures the active account in-app (#360) — a button + optional label field that sends the verb over the #358 control socket and renders an honest pending → done → error from the redacted ack (no credential ever reaches the client). It is an onboarding affordance: a populated panel carries no capture bar, so the steady state stays a read-only glance and adding an account lives off-panel. The captured row then arrives on its own via the live watch stream — the affordance never inserts it. To add a different account the operator runs claude /login first, then captures (capture snapshots the active account; it is not a picker). Only where the app genuinely can’t act does it still copy a command: version-skew’s Copy update command (brew upgrade sessiometer). daemon-starting stays a static forming glyph — the app fakes no progress it isn’t doing.
  • Empty-roster is distinct from daemon-down. First-run uses a calm accent-tinted add mark and inviting copy — unmistakable from the broken-link disconnected shape or the fault triangle.
  • Provider-neutral, minimal identifiers, every state. Wherever accounts render, each account gets a per-account IDENTITY colour + smart monogram (a generic, low-chroma disambiguation cue — #445 below) and a plain text label, never a brand colour or logo. The operator-chosen label leads and the secondary line is provider-only; with no custom label an account's label defaults to its account UUID (the roster stores no email — it lives only in the encrypted credential). The footer swap-history now reads by alias (Work → Personal).
  • Same-local-part accounts stay distinguishable (#445). A roster of accounts sharing a local-part (work-alice, work-bob, …) would collapse to one badge letter under label.first + tail-truncation. The disambiguation kit restores legibility with THREE cues, none alone sufficient: a per-account colour (a fixed low-chroma palette, accent hue excluded, seeded client-side from the wire label — the one identity handle, R-2; there is no account_uuid on the wire), a smart 2-char monogram from the label's distinguishing token (collision-escalating, so two similar labels never share a pair), and middle-truncation (not tail) so the distinguishing suffix / domain survives elision. The colour is never colour-alone — always paired with the monogram glyph + the label text (WCAG 1.4.1) — and clears WCAG-AA in both themes (badge fill ≥3:1, monogram ≥4.5:1). The per-account colour is a deliberate deviation from this mock's former monochrome badge; a faithful real-popover screenshot vs this updated oracle (both themes) + an owner eyeball is the pre-lock check (render re-baseline is #440).
  • Per-row switch is discoverable, not loud (#448). Every switchable row (non-active, viable) carries a persistent quiet swap chip in its trailing slot — neutral --text-3 at rest so a first-time operator sees the row is actionable on a transient popover, brightening to --text-2 when the row is armed (hover / focus). It stays neutral, never accent: the single accent action is the footer Swap hero (Von Restorff). The chip only hints at rest — the wash + pointer cursor still gate arming, so the persistent mark aids discovery without raising accidental-swap risk (the daemon re-validates every target, and a swap is reversible). The active row and a dropped/paused roster carry no chip.
  • No-wrap by construction, every state. Aliases own their line and truncate; badges, tags, health glyphs, strip actions and swap buttons are fixed-width and can never be pushed off. Meters use a fixed 4-column grid so % and reset countdowns stay column-aligned.
  • Light + dark hold uniformly. Healthy is shown in both themes; degraded states in Light with Dark spot-checks (not-running, keychain-locked, empty-roster). Every degraded token — strips, fault/attention badges and the amber stale modifiers — is themed, so dark applies uniformly to all nine.
